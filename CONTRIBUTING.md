# Contributing to Starscape Orbit

Thanks for helping improve Orbit. The most useful issue reports treat the issue as evidence, not just a request. Share what you saw, what you expected, and enough context that someone else can reproduce or reason about it.

## Before You Open an Issue

- Search existing issues for the same bug, request, or discussion.
- Use the matching issue template when GitHub asks what kind of issue you want to open.
- Keep one issue focused on one problem or one feature request.
- Include exact commands, screenshots, logs, or small examples when they help explain the behavior.
- Mark uncertainty directly. It is fine to say "I am not sure if this is related, but..."

## Bug Reports

A good bug report answers these questions:

- What did you do?
- What happened?
- What did you expect to happen instead?
- Can you reproduce it? If yes, what are the smallest steps?
- What environment did it happen in?

Useful environment details include OS, Node version, package manager, Orbit version, install method, browser, and whether you were using the CLI, web UI, Docker runtime, MCP, or an agent.

## Feature Requests

A good feature request describes the workflow before the solution:

- Who needs this?
- What are they trying to do?
- What is awkward, slow, confusing, or impossible today?
- What outcome would make the request feel done?
- Are there examples from another tool, a screenshot, or a rough sketch that clarify the idea?

It is fine to propose a specific design. It is more important to explain the problem and the acceptance criteria so the final implementation can fit Orbit's local-first, agent-first model.
